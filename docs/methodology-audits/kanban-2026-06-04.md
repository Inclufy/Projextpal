## Practice violations (highest priority)

**WIP-LIMIT ENFORCEMENT VERDICT: ⚠️ WARN-ONLY / DETECT-AFTER-THE-FACT — NOT BLOCKING.**
The headline doctrine check fails. The pull action `KanbanCardViewSet.move` (`backend/kanban/views.py:235-280`) reassigns `card.column` (line 258) and `card.save()` (line 279) **unconditionally**. The only WIP logic (`views.py:266-271`) runs *after* the move is already committed in memory and merely writes a `WipLimitViolation` row — it never `raise`s, never returns 4xx, never reverts. There is no `clean()`, no DB constraint, no signal (confirmed: zero `ValidationError`/`raise`/`constraint` in the move path; no `signals.py`). A team member can pile unlimited cards into any column. The check also uses `>=` and counts pre-save, so it is off-by-one relative to the true post-move count (`views.py:266,269`).

Frontend mirrors this: `KanbanBoard.tsx:40-45` computes `isOverWip` only to **color the column header red** and stamp a destructive badge; the move chevrons (`KanbanBoard.tsx:61-62`) are never `disabled`, so the UI offers no blocking either. `KanbanWIPLimits.tsx:35-42` is a pure read-out (count, "Over limit!" badge, progress bar) + an editor for the number — visualization only.

Other top violations:
- **Policies not surfaced where they govern.** `WorkPolicy` (`models.py:205-227`) is a free-text CRUD list FK'd to `Project`, not to `KanbanColumn`. `KanbanWorkPolicies.tsx` renders a flat list on its own page; no entry/exit/DoD policy is shown at the column it governs on the board. The seed even *asserts* "WIP limits enforced" as a policy (`views.py:698`) — doctrine claimed, code contradicts it. ⚠️.
- **Class of Service absent.** No CoS/expedite/fixed-date field on `KanbanCard` (`models.py:62-115` has only `card_type` + `priority`). Swimlanes exist (`KanbanSwimlane`) and the seeder makes an "Expedite" lane (`views.py:637-639`), but the `move` action applies **no** differentiated policy to it — expedite does not bypass WIP, jump order, or get a different SLE. ❌ behaviourally.
- **Blocked time not captured.** `toggle_blocked` (`views.py:282-289`) flips `is_blocked` + sets a reason but records **no timestamp**; there is no blocked-duration field or history row. "Blocked time" metric is therefore uncomputable. ⚠️.

## Feature proposals + redesign (ranked — both directions)
1. **Enforce WIP at pull (BLOCKING)** — *weight: highest · value: high · effort: low.* In `move` (`views.py:235-280`), before reassigning `card.column`, compute `new_column.cards.exclude(pk=card.pk).count()` and if `>= wip_limit` return `Response({'detail': 'WIP limit reached'}, status=409)` **before** any save. Allow an explicit `expedite` swimlane bypass. Frontend: disable the forward chevron + show a tooltip when target is at limit (`KanbanBoard.tsx:50,62`); the existing `toast.error(err.detail)` (`KanbanBoard.tsx:24`) already surfaces it. Fixes Direction-A headline + Direction-B pull step. **No migration → no data-guardian gate.**
2. **Class of Service on the card + lane policy** — add `class_of_service` choices (standard/expedite/fixed-date/intangible) to `KanbanCard`; make expedite bypass WIP (paired with #1), sort to top, and flag SLE breach. Page: board lane styling + WorkItems filter. *high weight · med value · med effort.* **Needs migration → data-guardian backup warning (CLAUDE.md §9).**
3. **Column-scoped policies (redesign WorkPolicy)** — add nullable `column` FK (+ `policy_kind`: entry/exit/DoD/pull) to `WorkPolicy`; render the governing policy inline at each column header on the board. Turns ⚠️ into ✅ for "make policies explicit." *high weight · med value · med effort.* **Migration → backup warning.**
4. **Blocked-time capture** — add `blocked_at` (and a `BlockEvent`/history) so blocked duration + aging WIP become computable; feed a "blocked time" + "aging WIP" tile into FlowMetrics. *med weight · med value · low-med effort.* **Migration → backup warning.**
5. **Improvement-action + cadence model** — persist kaizen actions (title/owner/status/source-metric) and cadence events (replenishment/standup/review) so "Continuous Improvement" and feedback loops stop being read-only. *med weight · med value · med effort.* **Migration → backup warning.**

**Structural note:** the board today is a *card store with decoration*. The single highest-leverage redesign is converting the `move` endpoint from "record-then-allow" to "validate-then-commit" (#1) — every other gap is additive, but this one changes the board's nature from advisory to a real pull system.
