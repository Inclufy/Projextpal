## Tailoring-enforcement verdict (headline) вЂ” does PhaseMethodology drive behaviour?
**вќЊ NO. The methodology label is purely decorative.** `PhaseMethodology.methodology` (`models.py:56`) is stored, serialized, and rendered as a `<Badge>` вЂ” nothing branches on its value anywhere.

Evidence (exhaustive grep of `PhaseMethodology` + `methodology ==` across the repo):
- Backend: every reference is CRUD/serialize/admin. The only consumer beyond storage is `HybridDashboardView` (`views.py:147-151`), which does `.values('phase','methodology')` вЂ” a flat dump, no branching. - **Zero** `methodology ==` / strategy branches in `backend/hybrid/`. **Zero** imports of `scrum/`, `waterfall/`, `kanban/`, or `agile/` from `hybrid/` (grep returned empty). A "scrum phase" and a "waterfall phase" are byte-for-byte the same object with a different string.
- Frontend: methodology surfaces only as `<Badge>{p.methodology}</Badge>` in `HybridPhases.tsx:102`, `HybridConfiguration.tsx:125`, `HybridOverview.tsx:81`, `HybridTimeline.tsx:52`, `HybridTasks.tsx:144,169`. No conditional rendering, no methodology-specific component, no gate/sprint/flow UI.
- Sidebar: `AppSidebar.tsx:623-637` returns a **static, identical 6-item nav** (Overview/Phases/Tasks/Timeline/Artifacts/Configuration) for every hybrid project вЂ” it never reads phase methodology, so navigation cannot switch per phase.
- Validation gap: methodology is a free `CharField(max_length=50)` (`models.py:56`); the serializer (`serializers.py:19-23`) imposes **no choices**, and the two frontend `METHODOLOGIES` lists disagree (`HybridConfiguration.tsx:24` has 10 incl. `pmi/safe/msp`; `HybridPhases.tsx:25` has 7). Any string is accepted; nothing would consume a valid one anyway.

## Feature proposals + redesign (ranked вЂ” both directions)
1. **[FLAGSHIP REDESIGN] Make `PhaseMethodology` a real strategy (weightГ—valueГ—low-effort: HIGHГ—HIGHГ—MED).** Constrain `methodology` to `choices` matching real backends; on phase render, **mount the corresponding methodology's components/flows** (scrum в†’ sprint board/backlog/DoD; waterfall в†’ gate/sign-off; kanban в†’ WIP-limited flow) by deep-linking to or embedding the existing `scrum/`, `waterfall/`, `kanban/` backends. This is the unrealized core value proposition. Direction A+B. New: `PhaseGate`/strategy resolver; sidebar must read per-phase methodology instead of the static list at `AppSidebar.tsx:623`.
2. **Mixed-governance gates (weight HIGH, value HIGH, effort MED).** Add `hybrid_phase_gate` (FK `PhaseMethodology`, entry/exit criteria, approver, signed_off_at). Predictive phases require sign-off to advance; adaptive phases compute progress from a DoD checklist, not a free slider. Closes Mixed-governance вќЊ + the "mark complete = PATCH 100" hole (`HybridPhases.tsx:71-72`).
3. **Transition points (weight MED, value HIGH, effort LOW).** Add `entry_criteria`/`exit_criteria` JSON + `predecessor` FK to `PhaseMethodology` so a Design-gate в†’ Build-sprints hand-off is explicit and roll-up can show blocked transitions. Closes Transition вќЊ.
4. **Methodology-aware roll-up (weight MED, value MED, effort LOW).** Dashboard should roll up per-methodology status (gate state for predictive, sprint/velocity for adaptive) instead of a uniform `progress` average (`views.py:147-152`, `HybridOverview.tsx:45`). Depends on #1вЂ“#2.
5. **Validate & unify methodology values (weight LOW, value MED, effort LOW).** Add serializer `choices`; reconcile the two divergent frontend `METHODOLOGIES` arrays (`HybridConfiguration.tsx:24` vs `HybridPhases.tsx:25`). Prereq for #1.
