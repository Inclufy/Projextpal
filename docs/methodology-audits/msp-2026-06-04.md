## Theme violations (highest priority) — measurable-benefits verdict first
1. **Measurable benefits — ⚠️→❌ in practice.** Schema *looks* faithful: `target_value`, `baseline_date`, `target_date`, `owner`, `measurement_method` (`models.py:20-25`) + a `BenefitRealization` time-series (`models.py:37-52`) + `total_realized` aggregate (`serializers.py:21-24`). But **no baseline_value field** (only a baseline *date* — you can record when, never the from-what), so variance-vs-baseline is unmeasurable. No realization cadence/review is enforced (no `@action`, no scheduled measurement). **Benefits are a measurable-looking list, not a managed profile.**
2. **❌ The MSP module is dead code from the UI.** `ProgramBenefits.tsx:19` fetches `/programs/{id}/benefits/`, which resolves to `programs.ProgramBenefitViewSet` (`programs/urls.py:48`) backed by a **separate `programs.ProgramBenefit` model** (`programs/models.py:164`). The MSP endpoint `/api/v1/msp/programs/{id}/benefits/` (`core/urls.py:63`) is **never called by any page**. Two parallel benefit models exist; the UI uses the non-MSP one. Field shapes even diverge (`ProgramBenefits.tsx:41` expects `type`/`realized_value`/`on_track`; MSP serializer emits `status`=identified/planned/realizing/realized + `total_realized`). MSP benefits would render blank.
3. **❌ No Vision, no Blueprint (POTI).** No model, no real page. Sidebar `Blueprint` route → `ProgramDashboard` (`App.tsx:670-674`). The future-state target operating model that the whole programme builds toward does not exist.
4. **❌ No Business Change Manager role / no transition.** Sidebar `Transitions` → `ProgramRoadmap` (`App.tsx:690-694`). No construct embeds delivered capability into operations — MSP's core engine is absent.
5. **❌ No transformational-flow stage gating.** Tranche `status` is a free label (`models.py:58-63`); nothing gates tranche→tranche on a boundary review, and benefits/tranches/projects don't roll up.
6. **⚠️ Access-check bug.** `_verify_program_access` queries `Project.objects.filter(id=program_id...)` (`views.py:18`) but the FK and `get_queryset` filter on `Program` (`views.py:33`). Verification runs against the wrong table; likely always denies or mis-scopes on the nested create path.
