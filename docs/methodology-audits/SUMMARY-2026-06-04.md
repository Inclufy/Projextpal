# Methodology Suite — Consolidated Analysis & Academy Training Map

**Date:** 2026-06-04 · **Scope:** 13 methodologies (7 project-level, 5 program-level, 1 governance layer) × product fidelity (A/B audits) × Academy training coverage.
**Source:** per-method reports in `docs/methodology-audits/*.md`; Academy inventory in `frontend/src/data/academy/courses/`.

---

## 1. The one-line thesis

Across all 13 modules the data is broad and the **enforcement is thin**. The schema almost always stores the right artifacts; what's missing is *behaviour* — gates that block, decisions that flow downstream, configs that change how the system runs. **Modules with `0` enforced `@action`s are CRUD record-stores, not methodologies.** And — critically — **the same six modules that are hollow in the product also have no real training in the Academy.** Product maturity and training maturity are correlated; the exceptions (SAFe especially) are the dangerous ones.

## 2. Per-method summary — what works, what's missing

### Project-level

**Scrum — fidelity ~41%.**
Works: rich model set (~15 models, ~20 routes), backlog/sprint/daily/retro all present.
Missing: **Definition of Done doesn't gate the Increment** — `BacklogItem.update_status` accepts `done` with no DoD check (`scrum/views.py:186`); the `DoDChecklistEntry`/`Completion` models are orphaned (no viewset, no UI). No Product Goal. Sprint Planning is a form, not a PBI-selection + goal-commit event.
Fix priority: wire the DoD checklist into a per-sprint gate that blocks `→done`.

**Kanban — fidelity ~56% (highest project-level).**
Works: flow metrics (~69%), board, swimlanes, classes of service.
Missing: **WIP limit is warn-only** — `move` (`kanban/views.py:258`) commits the card then logs a `WipLimitViolation` *after the fact*; UI chevrons never disable. It's an advisory board, not a pull system.
Fix priority: convert `move` to validate-then-commit (409 at limit, explicit expedite bypass).

**Waterfall — fidelity ~38%.**
Works: Testing module has genuine requirement→test traceability + CPM critical-path on the Gantt.
Missing: all three headline controls fail — **phase gate is advisory** (`complete` ignores sign-off, `views.py:222`), **baselines are mutable** (Edit/Delete; CR-approve never re-baselines), **EVM is dead code** (correct CPI/SPI properties on `WaterfallBudget.models.py:457` + an `EarnedValueRecord` table, but no serializer/view/URL/UI exposes any of it).
Fix priority: enforce the phase gate (block `complete` until deliverables approved) + immutable baselines + surface EVM.

**Agile — fidelity ~57%.**
Works: Vision→Goal tracing, personas, budget.
Missing: **~80% redundant Scrum clone** with weaker enforcement (scrum is richer). DoD never gates "done" (`agile/views.py:376`); assignee never surfaced in any agile UI (0 grep hits); Goal→Epic→Item chain doesn't trace.
Fix priority: **differentiate, don't consolidate** (a live customer uses these routes) — lean Agile toward Manifesto-generic flow metrics + outcome tracking distinct from Scrum's ceremonies.

**LSS Green Belt — fidelity ~37%.**
Surprise: real DMAIC governance lives in a *shared* `backend/sixsigma/` app (17 models, SPC compute, charter/tollgate/closure actions) wired alongside the thin `lss_green` app.
Missing: **tollgate `approve()` is an unconditional rubber-stamp** (`sixsigma/views.py:670`) — never verifies deliverables, never blocks next phase. Cp/Cpk never computed (manual entry only). Three duplicate metric stores.
Fix priority: add a `can_pass()` precondition to `TollgateReview.approve` + a `compute_capability` action + collapse duplicate metric stores onto `sixsigma`.

**LSS Black Belt — fidelity ~46%.**
Good news: does **not** float free — reuses Green's `lss_green.DMAICPhase` (`lss_black/models.py:172`) and the shared `sixsigma` spine (MSA/Gage R&R, FMEA/RPN, savings/ROI).
Missing: tollgate still a record not a gate; the `lss_black` advanced tools (HypothesisTest/DOE/SPC/ControlPlan) are isolated CRUD with no FK to phase/root-cause/solution; duplicate `lss_black` vs `sixsigma` Hypothesis/SPC/ControlPlan backends + duplicate sidebar item (`AppSidebar.tsx:449` vs `:492`).
Fix priority: enforce the gate, FK advanced tools to the decision spine, de-duplicate the parallel backend.

**Hybrid — fidelity ~18% (lowest project-level).**
Missing: **`PhaseMethodology` is purely decorative** — `methodology` is a free CharField (`hybrid/models.py:56`) rendered only as a Badge; zero `methodology ==` branches anywhere, zero imports of scrum/waterfall/kanban from `hybrid/`. A "scrum phase" and a "waterfall phase" are the same flat-task object with a different string. The module's entire value proposition is unrealized.
Fix priority: make `PhaseMethodology` a real strategy that mounts the per-methodology backends per phase.

### Program-level

**SAFe — fidelity ~14%.**
Missing: no PI loop. 4 flat models, 0 actions; all 7 SAFe sidebar routes render the *same* placeholder `ProgramDashboard` (`App.tsx:633`) — the `safe/` API has **zero frontend consumers**. An RTE can't run a PI; a team has nothing to pull. No features/stories, no team linkage, no WSJF, no System Demo, no I&A.
Fix priority: build the real PI loop (Feature/Story/Dependency/SystemDemo/I&A models + WSJF + BV roll-up) and replace the shared-shell routes.

**MSP — fidelity ~14%.**
Missing: benefits schema looks faithful (target/owner/baseline_date + `BenefitRealization` time-series) but has no `baseline_value` (variance uncomputable), no cadence, and is **orphaned** — `ProgramBenefits.tsx:19` calls a *different* `programs.ProgramBenefit` model. MSP benefits are dead code from the UI. No Vision, no Blueprint, no benefits map, no tranche loop.
Fix priority: unify benefits onto `msp.MSPBenefit`, add `baseline_value`, wire the frontend; then Blueprint(POTI) + tranche gates.

**PMI — fidelity ~9% (lowest overall).**
Missing: a flat component list + a meeting-minutes log. `PMIGovernanceBoard.decisions` is a free JSON blob (`models.py:61`) with no component FK; the board authorizes/stops nothing; component status freely PATCHable. The entire `/api/v1/pmi/` backend is **stranded** (frontend uses the `programs` app). No charter, roadmap, benefits, stakeholder engagement.
Fix priority: `PMIGateDecision` (board→component) + `authorize_component`/`stop_component` actions as the only path to mutate component status; wire to the UI.

**P2-Programme — fidelity ~8%.**
Missing: 2 models, 0 actions, **zero frontend consumers** (sidebar links generic `programs` routes). Blueprint stores POTI as free text but drives nothing and isn't required; no programme board, no authorization gate (a "proposed" project PATCHes straight to "active"). Plus a tenant-scope bug — `_verify_programme_access` checks `projects.Project` while the FK is `programs.Program` (`views.py:5` vs `models.py:17`).
Fix priority: `P2ProgrammeBoardDecision` + `authorize`/`stop` actions as the only status-advance path; fix the scope bug; coordinate Blueprint/benefits with MSP.

**Hybrid-Programme — fidelity ~13%.**
Missing: **config fully decorative** — repo-wide grep for `primary_framework`/`secondary_frameworks`/`governance_structure` returns zero consumers; 0 actions; no frontend references; sidebar even hardcodes `programMethodology = 'hybrid'` (`AppSidebar.tsx:966`). Two contrasting configs are indistinguishable at runtime.
Fix priority: a `ConstituentAuthorization` model whose `authorize` action branches on the config (predictive→stage-gate, adaptive→cadence checkpoint).

### Governance layer

**Governance — fidelity ~38%.**
Works: stakeholder power/interest matrix is genuinely implemented (`Stakeholders.tsx:132`); cross-tenant `get_queryset` leaks are fixed.
Missing: **a Decision does nothing downstream** — stores status/impact/evidence, FKs *up* to program/board/meeting, but has no downstream-authorization FK, no signal/action; approving a decision never changes a Project/Program/Portfolio state. Meetings hold free-text agenda/minutes with **no MeetingAction model** (no owners/due dates/closure). Portfolio doesn't prioritize/fund. Governance is theatre.
Fix priority: wire Decision→component authorization (`outcome` + target FK + `apply` action + immutable audit row) + a MeetingAction lifecycle model.

## 3. Cross-cutting patterns (appear in ≥3 modules)

1. **Gate-as-record, not gate-as-block.** Tollgates, phase gates, WIP limits, DoD, board decisions all *record* but don't *enforce*. (scrum, kanban, waterfall, lss-green, lss-black, pmi, governance)
2. **Stranded backends.** Fully built program APIs with zero frontend consumers because the UI routes to the generic `programs` app instead. (safe, msp, pmi, p2-programme, hybrid-programme)
3. **Decorative configuration.** A stored label/config that nothing branches on. (hybrid `PhaseMethodology`, hybrid-programme `HybridGovernanceConfig`)
4. **Duplicate/parallel stores.** Same concept implemented twice. (lss `sixsigma` vs `lss_green`/`lss_black`; msp vs `programs` benefits; agile vs scrum)
5. **Computed properties never surfaced.** Correct formulas exist but no serializer/UI exposes them. (waterfall EVM, lss capability)

**Academy courses with no matching product methodology** (generic / supporting): `pmFundamentalsCourse` (5/26), `msProjectCourse` (3/18), `leadershipCourse` (3/16, 8375 lines — deepest), `stakeholderManagementCourse` (3/14), `aiLiteracyCourse` (4/16), `programManagementCourse` (3/16, generic program). Stubs with no content: `risk-management`, `pm-leadership`, plus the four methodology stubs above.

### What the map tells us
- **Strong correlation:** the 7 well-built product methodologies all have full courses; the 6 hollow ones have only stubs. Training investment tracked product investment.
- **The dangerous exception is SAFe** — a full 16-hour course (PI Planning, ART, System Demo, I&A) sits on top of a 14%-fidelity product where those screens are placeholders. A learner finishes the course, opens the app, and the PI loop they were taught **doesn't exist**. Highest training-credibility risk.
- **LSS Black Belt** is the inverse: the *product* has Black-Belt tooling (HypothesisTest/DOE/SPC/ControlPlan) but the Academy stops at Green Belt — a missed upsell and a coverage gap for the more advanced practitioner.
- **Duplicate SAFe catalog entry** (`safeCourse` real + `safe-fundamentals` stub) should be reconciled so the marketplace shows one canonical SAFe course.

## 6. Caveats
- Fidelity %s are reviewer estimates from a read-only static audit (models + views + pages + grep), not runtime measurements.
- "0 `@action`s" counts DRF `@action` decorators only; some enforcement could live in serializers/signals — each report checked, but treat as directional.
- No production was touched; no migrations run. Every redesign that adds a model carries the mandatory data-guardian backup gate (CLAUDE.md §9).
